# Action Steps — User Comments

## 1. Can't upload PDF/Word
Current build (`static/index.html`) only accepts `.txt` ("PDF parsing isn't wired up yet"). Older prototype (`app.js`) parses PDF but never supported `.docx`.
- Wire up PDF parsing in the current build (reuse old `extractTextFromPDF` logic).
- Add `.docx` parsing (e.g. `mammoth.js` client-side, or `python-docx` server-side).
- Update the `accept` attribute and hint text once both work.

## 2. Privacy concern about CV data
CV text leaves the device and goes to Claude's API for classification (flagged in `engineering.md` as a real privacy regression for this user base, given visa/legal sensitivity). Not persisted or logged, but never disclosed to users.
- Add a visible, plain-language privacy notice on the upload screen: what happens to the text, that it's discarded after the request, who sees it.
- Consider stripping name/contact info before sending to the LLM.

## 3. "Try-it sprint" isn't understood
By design — `ui.md` rules out any onboarding/explainer beyond the one-line header tagline. Not holding up against real users.
- Add one explanatory sentence wherever the term first appears (the "Start my 1-week try-it sprint" button, the sprint screen) — e.g. "a day-by-day hands-on trial task, not a class or meeting."

## 4. Only 4–5 titles regardless of background
`app/categories.py` is a closed, hand-curated set of 5 fields (M&E, NLP/Data Science, Web Dev, Digital Health, Business Ops). Every CV gets forced into the best of these 5, even a poor match. Already an open question in `product.md`.
- Expand the category library.
- Add a low-confidence "this may not be a strong fit" path instead of silently forcing a match.

## 5. Same sprint for everyone in a role
Sprint, skills, and timeline content is static and hand-authored per category — identical for every user matched to it. Intentional v1 scope, but flagged in the design doc's handoff as worth revisiting.
- Lightweight fix: use the evidence quotes already extracted from each CV to swap in 1–2 personalized sprint days instead of a fully static list.

## 6. Bilingual support and simple language for Myanmar users
This product is for Myanmar users. The current UI is English-only with no language toggle.
- Add a language switcher with at least 2 languages (e.g. English and Burmese).
- Simplify UI copy and sprint/result text so it's easy to understand for non-native English speakers.

## 7. AI-powered career matching system using public career-seeker data
Build a system that collects publicly available career seeker data from open-access websites.
- Identify open-access sources of career seeker data (e.g. public profiles, job boards) and define what's lawful and ethical to collect.
- Build a data collection pipeline (scraper/ingestion) with clear scope, refresh cadence, and storage plan.

**Implemented (this build), with a deliberate scope change — see below.** Scraping public profiles or job boards for data about identifiable career seekers raises real privacy/legal exposure (the same concern flagged in #2, and worse here because it's about third parties who never consented). Built instead on a public *occupational title taxonomy* — no data about any individual, ever:
- Source: Hugging Face dataset [`gpriday/job-titles`](https://huggingface.co/datasets/gpriday/job-titles) (CC-BY-4.0), built from ESCO v1.2.0, the O*NET Database v29.3, and OSCA — generic job titles, not people.
- `data/job_titles_sample.json` — ~1,000 deduplicated titles gathered via the dataset's public `/search` and `/rows` REST endpoints, bundled in this repo so the matcher works offline.
- `scripts/fetch_job_titles.py` — stdlib-only (`urllib`, no new dependencies) ingestion script that pages through the full ~65,248-row dataset for production use. Not runnable in this build's sandbox (its network access is allowlisted and excludes huggingface.co); run it from a normal environment to refresh `data/job_titles_full.json`.
- The only "collection pipeline" input at request time is the user's own CV text, submitted directly by them — same consent model as `/api/classify` already has.

## 8. Skills/experience pattern matching and job recommendations
Using the system in #7, analyze skills and experience via pattern matching and recommend best-fit job positions for each candidate.
- Define the pattern-matching/matching logic (rules-based, embeddings, or LLM-based) against collected candidate data and open job postings.
- Output a ranked or best-fit job recommendation per candidate, with a clear explanation of why each match was made.

**Implemented (this build).** `app/job_title_matcher.py` — hand-rolled TF-IDF + cosine similarity (pure stdlib + numpy, no scikit-learn) between the CV text and every title in the corpus from #7. Wired into a new `POST /api/match-titles` endpoint (`app/models.py`, `app/main.py`), parallel to `/api/classify` but unbounded by the 5-category closed set, addressing comment #4 ("only 4–5 titles regardless of background") with a much wider candidate pool.
- Each result includes `matched_terms` — the actual overlapping words between the CV and that title — as the "why this match" explanation, in the same spirit as `evidence_quotes` in `/api/classify`.
- Tested in `tests/test_job_title_matcher.py` (9 tests, all passing under this sandbox's Python) and `tests/test_api.py` (HTTP-level tests, written but not executable in this sandbox — see limitation below).
- **Known limitation #1 — sandbox-sampled corpus.** `data/job_titles_sample.json` is ~1,000 of the dataset's ~65,248 titles, gathered via keyword search rather than a full download (this build's sandbox can reach huggingface.co only through a separate fetch tool, not through Python/pip). Run `scripts/fetch_job_titles.py` in a normal environment and point `app/job_title_matcher.DATA_PATH` at the result for full coverage.
- **Known limitation #2 — not live-tested end-to-end.** This sandbox's Python (3.10) can't run the rest of this backend (`pydantic`/`fastapi`/`anthropic` are built for Python 3.12, this project's declared minimum) — a pre-existing, unrelated environment issue, not something this feature introduced. The core matching logic was verified directly; the new endpoint code mirrors `/api/classify`'s exact pattern and should be smoke-tested with `uv run pytest` / `uv run uvicorn app.main:app` in your own environment before shipping.
- **Known limitation #3 — TF-IDF favors rare words.** A CV word that's common across many titles (e.g. "developer") can be outranked by a rarer incidental word (e.g. "building", as in "building dashboards") if the rarer word happens to repeat. Fine for a v1 prototype; worth revisiting with a smarter scoring scheme (e.g. sublinear term-frequency scaling, or an embedding-based matcher) if this becomes a primary surface.

## 9. Make open-corpus matching the only path, and generate sprints per match (closes #4, closes #5)

Explicit instruction for this build: *"in app/categories.py, update the sprints for all 991 open-corpus titles and you may use generate with Claude per match and forget everything about the 5 categories."*

**Implemented (this build).** The closed 5-category system from v0.2 is gone — `app/categories.CATEGORIES`, `POST /api/classify`, `app/classifier.merge_classification`/`validate_quotes`, and the category-specific Pydantic models (`Category`, `ClassifyRequest`/`ClassifyResponse`, `LLMCategoryResult`/`LLMClassification`) have all been removed rather than kept alongside the new flow, per the instruction to "forget everything about the 5 categories." There are now exactly two routes:

- `POST /api/match-titles` (unchanged from #8) is the only "find my title" step — full ~1,000-title open corpus, TF-IDF + cosine similarity, no LLM call. Now also returns `evidence_text` per match (built by the same `format_evidence_text` used for the old evidence quotes, just fed `matched_terms` instead).
- `POST /api/sprint` (new) — given a CV, the one title the user is currently looking at, and that title's `matched_terms`, asks Claude Sonnet 4.6 (`app/llm_client.generate_sprint`) to write the 2-3 examples, 7-day sprint, skills, and 3-phase timeline for that specific CV-title pairing. This directly answers #5 ("same sprint for everyone in a role") — content is no longer static or shared across users, since it's generated fresh per CV.

**Design choice worth flagging: generation is lazy, not batch.** Rather than generating sprint content for every ranked title up front, `/api/sprint` is called once, only for whichever single title is currently on screen — mirroring the product's existing "show one at a time, cycle on request" UX (`static/app.js`'s `fitNoBtn` handler). Generating sprints for all of, say, the top 10 matches when a user only ever looks at 1-2 in practice would mean ~5-10x the LLM cost and latency for no benefit. Flagging this since it wasn't explicitly specified — happy to switch to eager/batch generation if there's a reason to want it (e.g. pre-warming for a "compare titles side by side" view later).

**Fallback content, not a hard error, on generation failure.** `app/categories.py` now holds a single, title-agnostic `FALLBACK_SPRINT_CONTENT` dict (generic but usable 7-day sprint/skills/timeline/examples) instead of the 5 hand-authored categories. `POST /api/sprint` falls back to it — merged with the requested title — on *any* failure from `generate_sprint` (network, missing/invalid API key, or a malformed LLM response), returning 200 with generic content rather than a 502 on what is otherwise the product's main screen. This is a deliberate trade-off: it keeps the product usable end-to-end even without a configured `ANTHROPIC_API_KEY`, but means a generation failure is currently silent to the end user — no "this sprint is generic, not personalized" indicator. Worth adding before this has real users.

**Frontend updated to match (`static/app.js`, `static/index.html`).** The "find my title" flow now calls `/api/match-titles`, then awaits one `/api/sprint` call (cached on the match object, so revisiting the same title via the sprint/end screens never re-fetches) before showing the result screen. The "No — try my next-best title" cycling button does the same lazy fetch for the next-ranked title before rendering it. Copy referencing the old "five fields" framing and "quoted phrase" classification language has been updated to describe the open corpus + generated-sprint model instead.

**Tests rewritten:** `tests/test_categories.py` (now checks `FALLBACK_SPRINT_CONTENT`'s shape instead of the old `CATEGORIES` list), `tests/test_classifier.py` (kept only the `format_evidence_text` tests — `validate_quotes`/`merge_classification` tests removed along with those functions), `tests/test_llm_client.py` (rewritten for `build_sprint_prompt`/`generate_sprint` against a mocked Anthropic client, mirroring the old `classify_cv` test structure), `tests/test_api.py` (old `/api/classify` section removed; new `/api/sprint` section covers the happy path, the fallback-on-failure path, and validation errors). `tests/test_job_title_matcher.py` is untouched — that module didn't change.

- **Known limitation #4 — sandbox can't execute this.** Same pre-existing issue as limitation #2: this sandbox's Python (3.10) can't import `fastapi`/`pydantic`/`anthropic` (built for the project's declared 3.12+), so none of `app/models.py`, `app/main.py`, or `app/llm_client.py` could be run here. The pure, dependency-free logic (`format_evidence_text`, `FALLBACK_SPRINT_CONTENT`'s shape, `build_sprint_prompt`'s string construction) was checked directly in this sandbox; everything else was written by careful pattern-matching against the working v0.2 code and needs `uv run pytest` / `uv run uvicorn app.main:app` in a real environment before shipping.
- **Known limitation #5 — generated content has no equivalent to quote validation.** The old `validate_quotes` guaranteed every `evidence_quote` shown to a user was a real substring of their CV — a hard, mechanical safety net. Generated sprint content has no analogous check: a "2-3 sentence anonymized example" or a day-task could still be generic, slightly off-topic, or (rarely) factually confused in a way nothing in this pipeline would catch. Worth a manual content-quality pass against a handful of real CVs before relying on this for real users — the prompt asks for grounding in the CV and matched terms, but nothing currently verifies the model followed through.
- **Known limitation #6 — per-request LLM cost/latency, not cached server-side.** Every `/api/sprint` call is a live Claude request; the only caching is client-side, per-session, per-title (`static/app.js`). Two different users who both happen to land on the same title (or the same user across two sessions) regenerate from scratch. Fine at current scale; worth a server-side cache keyed on `(title, matched_terms)` if usage grows.

## 10. Drop per-request Claude generation — no budget for API costs (supersedes #9's sprint-generation half)

Explicit instruction for this build, verbatim: *"I can't afford API costs, it is an educational project, do you have any other suggestion."* This came after confirming (checked against Anthropic's own pricing/support docs) that a Claude Pro/Max/Cowork subscription does **not** cover Claude API/Console usage — those are billed separately, per token, and `app/llm_client.generate_sprint` from #9 was a live API call on every `/api/sprint` request. For a $0-budget educational project, that's a non-starter at any real usage volume.

Three options were on the table: (a) keep per-request generation and have the user pay for API credits directly, (b) generate all sprint content once, offline, in a single batch, then serve it statically forever, (c) hand-author a wider library of sprint content "archetypes" right now (covered by the chat session's own subscription, not the API) and match each of the 991 titles to its nearest archetype deterministically. The user picked **(c)**.

**Implemented (this build).** `app/llm_client.py` is deleted — there is no LLM, no network call, and no `ANTHROPIC_API_KEY` anywhere in this backend anymore, in production or in tests.

- `app/categories.py` now holds `ARCHETYPES` — 28 hand-authored sprint/skills/timeline/examples bundles, each with an `anchor_text` field (a list of real, related titles pulled from `data/job_titles_sample.json`'s actual vocabulary, not guessed) — plus `DEFAULT_ARCHETYPE`, a generic title-agnostic bundle used only when a title shares zero vocabulary with every archetype.
- `app/sprint_matcher.py` is new. It reuses `app/job_title_matcher.JobTitleCorpus` itself — the exact same hand-rolled TF-IDF + cosine-similarity machinery the title search already uses — except built over the 28 archetypes' `anchor_text` fields instead of the 991-title corpus. `assign_sprint_content(title)` vectorizes the matched title, finds the archetype with the highest cosine similarity, and returns its static content; `ArchetypeMatcher.best_match` falls back to `DEFAULT_ARCHETYPE` only on zero overlap.
- `app/models.SprintRequest` now carries only `title` — `cv_text` and `matched_terms` are gone, since archetype assignment never looks at the CV. `app/main.py`'s `/api/sprint` route calls `assign_sprint_content` directly; there's no failure path to catch anymore (no network, no parse step), so the old try/except-and-fall-back-to-generic-content logic from #9 is gone too — it always succeeds.
- `static/app.js`'s `ensureSprintContent` now posts just `{title}`; the footer note in `static/index.html` no longer claims sprints are "written by Claude... for this exact title and CV."
- `pyproject.toml` drops the `anthropic` and `python-dotenv` dependencies (nothing reads `.env` or talks to a model anymore); `.env.example` is deleted outright since there's no secret left to configure.
- Verified directly in this sandbox (numpy is importable here even though `pydantic`/`fastapi` aren't — see limitation #2/#4 above): ran the real 991-title corpus through `assign_sprint_content` and checked the archetype distribution. All 28 archetypes get used (smallest: 3 titles, largest: 102); only 6 of 991 titles fall to `DEFAULT_ARCHETYPE` (a handful of transport/operator titles and BLS "All Other" catch-alls that don't have a real white-collar-skewed analog among the 28 archetypes — falling back honestly here beats forcing a bad match).

**Trade-off, stated plainly — this is real personalization loss, not a free lunch.** Two CVs that both land on, say, "Junior Data Analyst" and "Data Engineer" will likely both map to the `data_scientist_ml` archetype and see byte-for-byte identical sprint content — there is no more CV-specific influence on sprint/skills/timeline at all (the brief CV-grounding #9's prompt asked for is gone along with the LLM call). This also re-opens, in a smaller form, the exact complaint #5 raised about the original 5-category system ("same sprint for everyone in a role") — now it's "same sprint for everyone in an archetype," just with ~28 buckets covering 991 titles instead of 5 buckets covering everything, which is a meaningfully better ratio but not true 1:1 personalization. `evidence_text` on the result screen is unaffected — that's still built deterministically from the CV's own matched terms (`app/classifier.format_evidence_text`) and was never part of this change.

- **Known limitation #7 — one corpus title (`M&E Officer`) misroutes due to the shared tokenizer dropping single-letter tokens.** `app/job_title_matcher.tokenize` (reused as-is by `sprint_matcher.py` for consistency with the title-search step) only keeps tokens of length >1, so "M&E" contributes nothing and "M&E Officer" matches purely on the generic word "officer" — it lands on `communications_officer` instead of the intended `monitoring_evaluation_officer`. Every non-abbreviated form in the corpus ("Monitoring and Evaluation Officer/Specialist", "Water Monitoring Officer", "Election Monitor", etc.) routes correctly; this is a single literal-abbreviation edge case, not a systemic problem, and wasn't fixed by changing the shared tokenizer since that risks regressing the already-verified `/api/match-titles` behavior. Worth a small, targeted fix (e.g. a tokenizer exception for known abbreviations) if this exact title turns out to matter in practice.
- **Known limitation #8 — `uv.lock` is now stale.** This sandbox's network is allowlisted and blocks `uv`'s own Python-toolchain download (`uv lock` fails trying to fetch a CPython build from GitHub), so the lockfile still lists `anthropic`/`python-dotenv` even though `pyproject.toml` no longer does. Run `uv lock` in a normal environment before relying on a fresh `uv sync`.
- **Known limitation #9 — sandbox still can't execute the FastAPI layer.** Same pre-existing issue as limitations #2/#4: this sandbox's Python can't import `fastapi`/`pydantic` (3.12+ only), so `app/main.py` and `app/models.py` weren't run here. Everything reachable with plain Python + numpy was: `app/categories.py`'s 28 archetypes were checked programmatically for correct shape (sprint==7, timeline==3, examples in [2,3], non-empty skills, unique ids), and the new `app/sprint_matcher.py` was run directly against the full real 991-title corpus (see above) — not just unit-tested in isolation. `tests/test_api.py`'s new `/api/sprint` section and `tests/test_sprint_matcher.py` still need `uv run pytest` in a real environment to actually execute.

## 11. Fix the standalone HTML prototype — it was never wired to #9/#10's open-corpus rewrite

User report, verbatim: *"but when I try HTML prototype, why it only only five titles rotatively."* The FastAPI backend (`app/`, `static/`) was fully rebuilt in #9/#10 to use the open 991-title corpus and the 28 free archetypes, but `ProofPath_Prototype.html` (and its duplicate `ProofPath_Prototype 2.html`) is a separate, self-contained, single-file demo that runs entirely in the browser with no server — it was never touched by that rewrite. It still ran its primary "find my title" / "No — try next" cycling logic off the original 5 hardcoded `CATEGORIES` entries from the very first prototype. An earlier session had embedded the full 991-title corpus and a TF-IDF search into the file, but only wired it up as a decorative "Other related titles" sidebar next to the real result — the 5-category logic was still what actually drove the title shown and the cycling, which is why cycling only ever rotated through 5 titles.

Two fixes were possible: (a) port the open-corpus matching and all 28 archetypes into this standalone file's JS, or (b) have the user run the already-fixed FastAPI server instead. The user chose (a) — upgrade the demo file, no server required.

**Implemented.** Both `ProofPath_Prototype.html` and `ProofPath_Prototype 2.html` (kept byte-identical, as established in #8) now run the same matching architecture as the real backend, reimplemented in plain JS with zero dependencies:

- The 5-entry `CATEGORIES` array is gone. In its place: `ARCHETYPES` (28 entries) and `DEFAULT_ARCHETYPE`, generated directly from `app/categories.py` so the two codebases carry identical sprint/skills/timeline/examples content — not retyped by hand.
- The file's existing `buildJobTitleCorpus`/`vectorizeJobQuery` TF-IDF helpers (already built for the 991-title sidebar search) are now reused a second time: a small `ARCHETYPE_CORPUS` is built over the 28 archetypes' `anchor_text` strings, and a new `bestArchetypeMatch(title)` vectorizes a title against it and returns the archetype with the highest cosine similarity (falling back to `DEFAULT_ARCHETYPE` on zero overlap) — the exact algorithm as `app/sprint_matcher.ArchetypeMatcher`, just running client-side.
- `tryFindTitle()` now ranks the CV text against the full 991-title corpus directly (`searchJobTitles(JOB_TITLE_CORPUS, text, 10)`) instead of scoring against 5 categories. `ensureSprintContent(match)` attaches each ranked title's archetype-matched sprint content lazily (mirrors `static/app.js`). The old `scoreText`/`buildWhyText`/`escapeRegex` keyword-counting functions are gone; `formatEvidenceText(terms)` replaces `buildWhyText`, ported verbatim from `app/classifier.format_evidence_text`'s wording so both apps phrase "why this title" identically.
- The now-redundant "Other related titles — open job-title index" sidebar (`renderBroaderMatches`) is removed — the main result *is* the open-corpus match now, so a second copy of the same search next to it added nothing.
- Footer note and the upload-screen hint text were rewritten to describe what's actually happening (open 991-title index + 28 archetypes, runs fully in-browser) instead of the old "keyword pattern-matching demo, not a trained model" line, which was inaccurate even before this fix. The "Start my 1-2 week try-it sprint" button was also corrected to "1-week" to match the actual 7-day sprint content.

**Verified.** `node --check` confirms the rewritten `<script>` block is syntactically valid. A full simulated run (sample CVs through the actual exported functions, then a stubbed-DOM run of the entire IIFE via Node's `vm` module, clicking the real `findTitleBtn`/`startSprintBtn`/`finishSprintBtn`/`fitNoBtn` handlers) confirms: matching now returns up to 10 ranked titles per CV (was 5), each "No — try next" click advances to a genuinely different title with correctly-matched archetype content, and the "you've run out of titles" closing screen fires correctly at the end of the ranked list rather than after only 5 clicks. Sample B (data-science CV) ranked Data Scientist → Data Officer → Data Analyst → Marketing Data Scientist → Data Consultant → Data Specialist → Computer Scientist → Health Data Analyst → Marketing Data Analyst → E-learning Developer, 10 distinct titles, before the closing message appeared — confirming the original bug is fixed. A title with zero archetype overlap still falls back to `DEFAULT_ARCHETYPE` (`general_role`) as expected. `grep` confirms no remaining references to the removed `CATEGORIES`/`scoreText`/`buildWhyText`/`renderBroaderMatches`/`escapeRegex`/sidebar-element IDs anywhere in either HTML file.

- **Note:** like #10's known limitation #7, the literal title "M&E Officer" still misroutes (to `communications_officer` instead of `monitoring_evaluation_officer`) here too, since this port reuses the same tokenizer behavior (drops single-character tokens) — not a new issue, just inherited from the backend it mirrors.
